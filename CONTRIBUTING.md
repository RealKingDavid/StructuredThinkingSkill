# Contributing to Structured Thinking Skill

Thank you for your interest in contributing! This skill helps people turn complex ideas into clear, actionable plans — and your contributions help make it better for everyone.

## Ways to Contribute

### 1. Improve Existing Frameworks

- Fix typos, grammar, or unclear language
- Add more specific subtasks to existing phases
- Improve time estimates with real-world data
- Add domain-specific examples (e.g., SaaS startup vs. food business)
- Update frameworks to reflect current best practices

### 2. Add New Idea Types

Have an idea type that doesn't fit the existing four categories? We welcome new reference files for:

- Social enterprise / nonprofit ventures
- Hardware / physical product development
- Research projects (academic or independent)
- Community or event organizing
- Career transitions or personal development plans

### 3. Report Issues

- A framework phase that doesn't make sense in order
- Missing edge cases (e.g., solo founder vs. team, no budget)
- Trigger conditions that are too narrow or too broad
- Subtasks that are too vague to be actionable

## Skill File Format

All reference files should follow this structure:

```markdown
# [Type] Framework

## Phases

### Phase 1: [Name]
[What this phase accomplishes]
- Tasks: [comma-separated list]
- Subtasks: [specific, actionable items]

### Phase 2: [Name]
...

## Key Risks to Surface
- [Risk 1]
- [Risk 2]
- [Risk 3]

## Type-specific notes
- [Contextual tip 1]
- [Contextual tip 2]
```

## Guidelines for Quality

### Do:
- Make subtasks specific and executable — "Write 5 problem hypothesis statements" not "Think about the problem"
- Include honest, ranged time estimates
- Surface real risks, not just generic warnings
- Consider both solo and team contexts
- Test the framework against a real idea before submitting

### Don't:
- Be vague or generic
- Duplicate advice already in SKILL.md
- Make phases too long (aim for 2–5 tasks per phase)
- Ignore edge cases (zero budget, solo, first-time founder, etc.)

## How to Submit

### For Small Changes (Typos, Minor Improvements)

1. Fork the repository
2. Make your changes
3. Submit a pull request with a clear description of what you changed and why

### For New Reference Files or Major Framework Changes

1. Open an issue first to discuss your idea
2. Get feedback before writing
3. Fork, write, and test thoroughly
4. Submit a pull request with sample inputs/outputs demonstrating the improvement

### Pull Request Format

- **Title**: Clear and descriptive (e.g., "Add nonprofit/social enterprise reference file")
- **Description**: What changed, why, and how you tested it
- **Examples**: Include a sample idea and the output your change would produce

## Code of Conduct

- Be respectful and constructive
- Focus on helping people move from idea to action
- Provide honest, accurate, and actionable frameworks
- Credit sources or inspirations when applicable

## Questions?

Open an issue with the "question" label and we'll help out.
