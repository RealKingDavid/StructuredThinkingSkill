# Product / UX Design Framework

## Phases

### Phase 1: Discover & Define
Understand users and frame the problem clearly before designing anything.
- Tasks: User research, competitive audit, define problem statement, align stakeholders
- Subtasks: Recruit 5+ research participants, conduct interviews or contextual inquiry, synthesize into themes, write "How Might We" statements, get stakeholder sign-off on problem scope

### Phase 2: Ideate & Concept
Generate and filter ideas before committing to a direction.
- Tasks: Brainstorm solutions, sketch concepts, prioritize ideas, define information architecture
- Subtasks: Run a design studio or solo sketching session, create 3+ distinct concepts, dot-vote or score against criteria, map user flows, define navigation structure

### Phase 3: Prototype
Build something testable without full engineering effort.
- Tasks: Create wireframes, build interactive prototype, define component inventory
- Subtasks: Lo-fi wireframes for key screens, hi-fi prototype for critical flows, annotate interactions, define responsive behavior, create a simple design system or component list

### Phase 4: Test & Validate
Put the design in front of real users and learn fast.
- Tasks: Plan usability study, run sessions, analyze results, iterate
- Subtasks: Write test script and tasks, recruit 5+ participants, observe and take notes, synthesize top findings, prioritize fixes by severity

### Phase 5: Handoff & Launch
Deliver a build-ready design and support implementation.
- Tasks: Finalize specs, document design decisions, support engineering, QA the build
- Subtasks: Annotate spacing, states, and edge cases; create a component library in Figma/equivalent; write interaction specs; conduct design QA on staging; define post-launch feedback loops

## Key Risks to Surface
- Skipping research and designing based on assumptions
- Too many concepts — failing to commit to a direction
- Prototype fidelity mismatches (too lo-fi to test real interactions)
- Handoff gaps — engineers making decisions the designer should have made
- Designing for the happy path without edge/error states

## Type-specific notes
- Always ask: "What is the core job to be done?" — center the design around that
- Accessibility is not a phase — it's built in from wireframes onward
- For mobile: thumb zones, gesture conventions, and offline states matter early
- For complex dashboards: information hierarchy and progressive disclosure are critical
